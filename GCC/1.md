# Why gcc is gcc?
## compile process
```
Frontend (C/C++/Fortran)
   ↓
GIMPLE（高层IR）
   ↓
SSA-based GIMPLE优化
   ↓
RTL（Register Transfer Language）
   ↓
Machine-dependent scheduling / register allocation
   ↓
Assembly
```
key point:
（1）双IR设计：GIMPLE + RTL
    - GIMPLE
    类似三地址SSA
    做高层优化（inlining、DCE、loop opt）
    - RTL
    更接近机器指令的表达（偏“寄存器级”）
    GCC独有，LLVM没有对应等价物
👉 本质问题：
    - GCC优化链条被“拆成两段IR体系”，中间转换复杂且语义分裂
    - 导致优化效果不统一（GIMPLE优化“跑不起来”在RTL上）
（2）后端强绑定 RTL
    后端优化很多直接操作 RTL
    instruction selection + scheduling + RA 深度耦合
    导致：
    新架构 bring-up 成本高
        - pass 重用性弱
## cmp
| 维度      | GCC          | Clang/LLVM    |
| ------- | ------------ | ------------- |
| IR体系    | GIMPLE + RTL | LLVM IR + MIR |
| 架构      | monolithic   | modular       |
| backend | RTL耦合        | 分层IR          |
| 扩展性     | 较弱           | 很强            |
| 优化风格    | 经验驱动         | pipeline驱动    |
| 工程复杂度   | 高（历史包袱）      | 中等            |
| 工具链     | 弱            | 强             |
| 新架构支持   | 慢            | 快             |

## summary
GCC 是“RTL时代的机器级编译器”，LLVM 是“IR驱动的分层编译框架”。

# How about gcc's IR?
```
GIMPLE = 高层 SSA IR（接近 LLVM IR，但更弱）
RTL = 低层机器描述 IR（接近 LLVM MachineInstr + selection 结果的混合体）
```
## GIMPLE
- 特征
    - 基于 SSA（有临时变量）
    - 三地址码风格（类似 LLVM IR）
    - control flow graph 明确（basic block）
    - 强调“语义简化”（例如：没有 phi 节点）
    - 不支持复杂类型（例如：结构体）
- pass(target-independent optimization layer)
    - constant propagation
    - DCE
    - inlining
    - loop optimization（部分）
    - SCCP
    - PRE（部分）
## RTL(Register Transfer Language)
RTL ≈ “非常早期的 Machine IR + 模板化指令表达”
`(set (reg:SI r1) (plus:SI (reg:SI r2) (reg:SI r3)))`
- 特征
    - instruction-level representation
    - explicit register / memory operations
    - machine constraints embedded
- 包含
RTL 中已经包含：
    - register class
    - instruction pattern
    - addressing mode
    - machine mode（SI/DI/DF...）
- 后续
    - instruction selection
    - scheduling
    - peephole optimization
    - register allocation（reload）

## ASM
final instruction encoding
register assignment finalized
scheduling finalized
assembly emission

## gimple and rtl
GIMPLE 不是“唯一 IR”，LLVM IR 是“唯一 IR”。
后面还有 RTL（而且很重）
很多优化在 RTL 才发生。。。。
- LLVM IR 更复杂
    LLVM IR 更像“虚拟ISA”
    GIMPLE 更像“简化后的中间表示”
- GIMPLE不承担任何的backend语义，只负责“语义简化”
但 GIMPLE 完全不关心 target
不包含：
    calling convention
    register constraint
    instruction mapping
### 如何理解 gcc 和 llvm 中间表达的区别？
GIMPLE 是“带有更多高级语言类型残余、且尚未完全物理化的 LLVM IR”。
RTL 不是 DAG，它是“已经初步完成了指令匹配，但还没分配好物理寄存器的伪汇编代码”。

未加工的 RTL 就是一个**“已经选好了所有舞步，但还没决定具体由哪位舞蹈演员站在哪个位置”**的舞台排练脚本。它完成了从“做什么”到“怎么做”的跨越，但还没解决“用谁做”的资源冲突。

## 优化链条
GIMPLE
 ↓
expand（= 指令选择）
 ↓
RTL（粗糙）
 ↓
combine（补 instruction selection）
 ↓
RA（reload）
 ↓
peephole + scheduling

所以实际上 GIMPLE->RTL 做完了大部分的指令选择，combine后的指令选择只是为了匹配合并优化后的pattern。

## DSP 编译器的gcc注意点
pattern 设计、寄存器/约束建模、调度，以及 loop 优化
### 喂奶
1. 关键pattern的保留，类似tensor operation
2. 控制优化“不过度”
    GCC 中端有些优化会：
    打散表达式
    引入临时变量
### gimple expand
1. optab 映射设计
2. define_expand
3. addressing mode 提前建模（提前计算addr）
### md pattern 设计（关键）
1. pattern 要覆盖“真实硬件能力”
你需要定义：
    MAC
    SIMD op
    saturating op
    特殊 load/store

2. operand constraint 非常重要
DSP 常见：
某些指令只能用特定寄存器
accumulator 专用寄存器
👉 constraint 写不好：
    RA 会爆炸（寄存器分配失败）
    reload 会插一堆垃圾代码 （寄存器重加载代码）

3. pattern 粒度要“偏大”
和 CPU 不同：
👉 DSP 更希望：
一条指令做更多事情

### combine pass：第二关键点
1. pattern 能被 combine 匹配
2. 避免 pattern 被破坏
### Register Allocation（DSP 最痛点之一）
1. 多寄存器类型
2. accumulator 特殊处理
3. spill 成本极高

### Instruction Scheduling（DSP 最痛点之二）
1. pipeline model
    ```
    latency
    issue rate
    functional units
    ```
2. software pipelining

### Loop 优化（DSP 的生命线）
1. hardware loop
2. loop unrolling
3. memory access pattern

## 4 大痛点
⭐ 1. pattern 设计（.md）
决定你能不能用上 DSP 指令

⭐ 2. expand（GIMPLE → RTL）
决定 pattern 能不能被触发

⭐ 3. combine
决定能不能做 instruction fusion

⭐ 4. RA + scheduling
决定性能是否崩掉
