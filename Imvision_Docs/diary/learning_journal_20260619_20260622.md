## BatchRewriter & PatternMatcher 概念澄清

### Q1: BatchRewriter 的数据结构和方法是干什么的？

**数据结构**：三个字段——`graph_`（图引用）、`pendingRemovals_`（待删除节点集合）、`committed_`（防重复提交）。

**为什么攒着删**：`Resolve()` 是重操作（全图拓扑排序 + shape/type 推导，O(N+E)）。每删一个节点调一次是 O(n²)，攒着一起删只调一次。

**Commit 调用链**：`Commit()` → 遍历 pendingRemovals_ 逐个 `ReleaseNode` → 单次 `Resolve()`。不是事务，没有回滚。

### Q2: 攒下来的 node 是没 match 过的？expand 失败会被删？

**错**。`TryExpand` 里的 `current.nodes.erase()` 是回溯搜索的变量清理，删的是匹配绑定关系，不是图节点。BatchRewriter 攒的是**匹配成功后要替换掉的旧节点**。

### Q3: BatchRewriter 是给谁的？为什么要 rewrite？

**命名误会**。BatchRewriter 只做"批量删除 + 单次 Resolve"，没有智能 rewrite。真正的 rewrite 逻辑在调用方（如 `SliceFuse::RewriteSlicePair`）。它是**给算法制作者的工具类**，不是面向最终用户的。

### Q4: BatchRewriter 和 PatternMatcher 是否独立？

**完全独立**。两个头文件互不引用，没有代码耦合。只在 `slice_fuse.cpp` 中组合使用（先 match 再删旧节点）。

### Q5: PatternMatcher::Match 会遍历全图？

是。三步：1) pattern 中选边最多的节点作为种子 → 2) 遍历 `kernelNet` 所有节点，对每个做 `NodeCheck（dynamic_cast + Attr lambda）`→ 3) 对每个 candidate 递归 TryExpand。

O(N) 全图扫描，没有索引跳过。

## 文档结构与命名

### Q6: 为什么要删 Node？

新创建的替代算子已接管原算子的输入输出。旧算子不删就变成**僵尸节点**——还在节点列表里、还引用 tensor，但下游消费者已连到新算子。会导致后续 pass（内存分配、Codegen、拓扑排序）出问题。删节点是**图正确性要求**，不是优化。

### Q7: BatchRewriter 作用到底是什么？

就是**把清理代码从 6 行压成 1 行**：

```cpp
// 不用：std::vector + 逆序 ReleaseNode + 条件 Resolve（6 行）
// 用：rewriter.RemoveNode() + rewriter.Commit()（2 行，封装了上面全部逻辑）
```

不参与匹配、不参与改写、不操作边。

## 硬件知识

### Q8: DmaIn 是什么？

**DMA Input = DDR → NPU L1 的数据搬运操作**。类继承：`Node → Kernel → DmaKernel → DmaInKernel`。内部生成 `NpuDmaIn` 硬件层生成 DMA 指令。

在 SliceFuse 中的角色：数据流起点，DmaIn 搬入整张 NHWC 图像 → Slice(H) → Slice(W)。DmaIn 在图中稀有（一个输入 tensor 一个），旧代码以此为遍历锚点。

### Q9: SliceFuse pattern 里为什么有 DmaIn？

**正确性守卫**。约束是"DmaIn 的所有 children 必须全是 SliceKernel，不能混入 Conv2D 等其他消费者"。如果只匹配 HSlice→WSlice 会漏掉这个约束。

## PermuteReplaceReshape 重构

### Q10: PermuteReplaceReshape 是干什么的？

把某些 Reshape 替换成 Permute（+ 辅助 Reshape），因为 MTE 硬件上 Permute 执行效率更高。Reshape 在框架层零成本，但 NPU 硬件上维度重排需要实际搬运。三种路径：简单 2-Permute、Spec 1-Reshape+1-Permute、复杂 4-Permute 链。

### Q11: 除了 PatternMatcher 还有什么重构点？

两个有价值：1) 提取 `Classify()` 纯函数（可单元测试）、2) 魔法数字常量化（`32`/`16384`/`131072`/`30` → 命名常量）。已记录到 `PERMUTE_REPLACE_RESHAPE_REFACTOR.md` 第六章。

### Q12: 重构后影响功能与性能吗？

不影响。维度判断、优先级分发、Impl 逻辑一字节没动。PatternMatcher 遍历全图和旧 for 循环一样是 O(N)。BatchRewriter 也就少了几行手动管理代码。

## 批量重构 9 个 Pass

### Q13: 重构策略

所有 pass 都遵循相同模式：

```
旧：GraphViewer → for (idx : order) → dynamic_cast<T*> → 创建替代算子 → ReleaseNode + Resolve
新：PatternBuilder("PassName").MatchNode("x", NodeType<T>()).Build()
    → PatternMatcher.Match() → for (match : matches) → 改写 → BatchRewriter.RemoveNode()
    → BatchRewriter.Commit()
```

| Pass | 特点 | 状态 |
|------|------|------|
| SplitExp | 1 个 Impl，直接 ReleaseNode+Resolve | ✅ |
| SplitInv | 2 个 Impl（mode=0/1），各自 ReleaseNode+Resolve | ✅ |
| SplitSoftmax | 2 个路径（channel-split/标准），`SatisfySplitC()` 分派 | ✅ |
| SplitBaseNorm | 4 个 Impl（NormType map 分派），4 处 ReleaseNode+Resolve | ✅ |
| TilingBaseNorm | nodes_to_remove vec，条件性 tiling | ✅ |
| TilingSinCos | 匹配双类型（Sin∥Cos lambda），条件性 tiling | ✅ |
| ConvTranspose2d | inline 改写，直接 ReleaseNode+Resolve | ✅ |
| ConvTranspose2d2 | nodes_to_remove vec，stride switch 分支 | ✅ |
| SplitMatmul | 两轮 graph_viewer + 子图匹配（Permute→Matmul） | ⏸ 暂缓 |

### Q14: "循环内直接 ReleaseNode" 是什么？

```cpp
for (auto idx : order) {
    auto* op = dynamic_cast<T*>(net->GetOp(idx));
    // ... 创建替代算子 ...
    net->ReleaseNode(op->Index());  // ← 直接在循环里删，没攒 vec
}
```

和攒 vec 模式的区别：删除时机在迭代中途，多数没调 Resolve。图状态可能不一致（删了节点但拓扑没刷），但通过 tensor 连接新算子可以正常工作。

### Q15: SplitBaseNorm 之前为什么不重构？后来为什么改了？

**最初判断（错误）**：以为 channel-split 路径有独立的拓扑序依赖，且 BaseNorm 子类型太多。

**实际验证**：整个文件只有一个 graph_viewer（RunOnModule 里的），Impl 函数只创建 Reduce/Eltwise/BatchNorm，没有嵌套遍历。和 SplitExp/SplitInv 结构完全一样。当时被"多分支"吓到了没读代码。

### Q16: BatchRewriter 为什么暂不改？

Alan（负责人）说 GraphRewriter 还在迭代，暂时不要引用。PatternMatcher 部分 API 稳定可以保留。等 Alan 定型后，9 个文件的回退是纯机械操作——每个 pass 三行替换。

## 关键收获

1. **PatternMatcher + BatchRewriter 是两个独立工具**，只在 pass 中组合使用
2. **"攒着删 + 单次 Resolve"是图编译的正确模式**——不是优化，是避免 O(N) 次重遍历
3. **所有 lowering pass 的本质**：硬件不支持的算子 → 拆成支持的基本算子 → 删旧算子
4. **判断重构可行性要先读代码**，不能被"看起来复杂"吓到（SplitBaseNorm 教训）
5. **重构的价值分层**：PatternMatcher（匹配层，稳定可做）> BatchRewriter（清理层，等待定型）
