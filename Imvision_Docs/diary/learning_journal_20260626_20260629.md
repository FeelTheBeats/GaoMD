### Q1: 原来的 10 个 split pass 可以从 main.cpp 删掉了吗？

**结论**：可以。`ComplexOpSplit` 已将 10 个 pass 的逻辑整合到一个 Visitor 中，一次图遍历完成所有分派。删掉 10 行 `enable_pass.push_back`，替换为 1 行 `ComplexOpSplit`。但各个 `.cpp` 中的 `PM_REGISTER_PASS` 暂时保留，仍可通过命令行单独调用。

**建议**：以后在这种"合并→删旧"的重构中，可以顺手建一个 cleanup task 清单，避免遗留尾巴。

### Q2: 删了 main.cpp 的注册后，pass 还能单独用吗？passes.h 的声明还要保留吗？

**发现**：三个层次各自独立——`passes.h` 是类声明（"这个类存在"），`PM_REGISTER_PASS` 是注册（"这个名字可被查到"），`main.cpp` 是执行流水线（"哪些 pass 按序跑"）。删掉 main.cpp 的注册只是不默认执行，但命令行仍可通过 name 单独调用。

后来 mentor 建议全删。最终决定：删除全部死代码——`PASS_DEFINITION`、`RunOnModule`、`PM_REGISTER_PASS`。

**评价**：你一开始的犹豫（"将来可能需要单独调某个 pass"）是典型的过度设计。mentor 全删的建议是对的——调试时注释掉 ComplexOpSplit 里的 Visit 分支即可，不需要维护 10 个废弃 pass 类。

**建议**：YAGNI 原则要牢记——不被调用的代码就是负债，不是资产。

### Q3: RunOnModule 删除后，函数功能还完整吗？和新的 SplitVisitor 是否等价？

**发现**：逐个对比 10 个原始 `RunOnModule` 和 `SplitVisitor::Visit`，发现了三个问题：

1. **`MergePermuteMatmul` 遗漏**：原始 `MatmulSplit::RunOnModule` 先调 `MergePermuteMatmul`（全局 permute-matmul 合并），再调 `PerformMatmulConverison`。但 `SplitVisitor::Visit(Matmul&)` 只调了 `ConvertMatmul`，丢了前置优化。
2. **节点双重移除**：7 个 impl 函数内部做 `ReleaseNode + Resolve`，`SplitVisitor` 又通过 `BatchRewriter::RemoveNode + Commit()` 再做一次。index 复用后可能误删新节点。
3. **`(void)` 丢弃错误**：所有 impl 返回值被 `(void)` 静默忽略。

**评价**：这个对比分析是你当天最深入的技术工作。不是简单确认"删了没事"，而是逐个追踪调用链，找出了原 commit 的 bug。这种"不相信表面结论，必须亲自验证"的态度值得保持。

**建议**：以后做类似对比时，可以做一个 checklist 表格（类型/node removal/错误处理/特殊逻辑），逐项打勾，不容易漏。

### Q4: extern → #include 头文件重构的意义是什么？

**核心**：`extern` 是跨 TU 的弱契约——签名改了不报错，链接时才炸。头文件 + `#include` 是编译期强契约——签名对不上当场报错。Alan 的建议（不要 extern，直接 include）是 C++ 基本实践。

最终为 10 个 impl `.cpp` 各创建了 `.h`，放到 `include/aic/transforms/split_op/` 下。

**评价**：一开始用 `extern` 作为快速原型手段是可以接受的。一旦跑通就应该升级到规范方式。你理解了这个过渡逻辑并执行了升级。

**建议**：`extern` 声明可以当作"TODO: extract header"的标记。写代码时只要用了 extern，就应该预期后续要建头文件。

### Q5: SplitVisitor 为什么要独立成文件？

**原因**：职责分离——`SplitVisitor` 是"怎么分派到每个 Op 的 impl"（dispatch 层），`ComplexOpSplit` 是"什么时候跑、跑完做什么"（编排层）。分开后 SplitVisitor 可被其他 pass 复用，ComplexOpSplit 也更干净。

**评价**：好的架构直觉。你在写代码之前先想"谁负责什么"，而不是把所有东西塞一个文件。

### Q6: BNInfo 的 ODR 冲突是怎么回事？是我的改动吗？

**发现**：`BNInfo` 在 6 个头文件中各自定义——`base_norm.h` 和 `softmax.h` 都在 `aic` 命名空间下定义了相同的 `struct BNInfo`，用 `#ifndef AIC_BNINFO_DEFINED` guard 防止编译报错，但这是创可贴。`401ee6da` 的 commit message 写了"Fix BNInfo redefinition"，实际加的就是这个 guard。

修复：提取到 `include/aic/ir/operators/bn_info.h`，两个头文件改为 `#include`。

**评价**：先确认是不是自己的改动（"这不是我引入的吧？"），确认后再修。这个顺序对——不碰不属于自己 scope 的遗留问题，但属于自己的就彻底修掉。

### Q8: FusedOp pattern-match 重写——Phase 2 的三个扩展真的需要吗？

**核心争论**：可行性文档把 `EdgeAttr`、`NodeTypeOr<T,U>`、`InputAttr`/`OutputAttr` 列为 Phase 2 必要扩展。但实际验证发现：

- `SliveFuse` 已经用 3 节点链证明了现有 API 够用
- `EdgeAttr` 可用 `Attr("from")` 内 `node.OutputNodesBegin()` 绕路
- `NodeTypeOr` 可用 `Attr()` 内 `CastNoCheck<T> || CastNoCheck<U>` 替代
- `InputAttr` 只是省一行 `Cast<Tensor>`

三项都是语法糖，不是功能缺口。

**评价**：这是你当天最体现"经验判断"的对话。文档说的是"需要"，你用自己的分析推翻了"需要"的结论，避免了过度设计。这种"看穿本质"的能力比写代码本身更难得。

**建议**：以后评审别人的设计文档时，同样可以用这个方法——先问"现有 API 能不能绕过去？"，再决定要不要加新东西。

### Q10: Visitor 模式重构的核心思路怎么讲？

**一句话版本**：将 10 个"遍历所有节点 → `dynamic_cast` → 调用 lowering"的独立 pass，合并为 1 个 pass + 1 个 Visitor，让 C++ 的类型系统替你做分派。

**技术内核**：Double Dispatch——`node->Accept(visitor)` 第一跳（虚函数）还原子类静态类型，第二跳（重载决议）精确匹配 `Visit(Exp&)`。`dynamic_cast` 从你的代码里消失了，编译器替你做。

**评价**：你能把技术点提炼成一两句话讲清楚，说明真正理解了。给别人讲技术的第一步永远是"一句话说清楚要解决什么问题"。
